# 專案建置計劃書

## 專案名稱
QueueFlow

## 文件版本
v1.0

## 專案類型
高併發線上候補排隊系統

## 一、專案背景

本專案旨在建置一套可支援高併發流量的線上候補排隊系統，模擬熱門活動、限量商品、餐廳候位或流量尖峰入口等情境。系統需在短時間內承受大量使用者同時進入，並維持排隊順序正確、狀態一致與通知即時。

本專案重點不在複雜商業流程，而在高併發系統常見議題，包括流量削峰、限流、快取設計、資料一致性、即時推播與事件化擴充。

## 二、專案目標

本專案之建置目標如下：

1. 建立可處理高流量請求之候補排隊系統。
2. 提供穩定的排隊順位管理機制。
3. 支援使用者即時查詢排隊狀態。
4. 提供管理端分批放行功能。
5. 建立逾時失效與取消機制。
6. 保留後續導入 Kafka 事件架構之擴充空間。

## 三、應用介紹

QueueFlow 為一套線上候補排隊系統，設計目的在於處理流量集中湧入時的排隊控管需求。當大量使用者同時進入系統時，平台需快速完成入列、排序、查詢與狀態更新，避免主系統直接承受流量衝擊。

此系統可應用於以下場景：

- 演唱會或活動候補入場
- 餐廳線上候位
- 門市限量商品排隊
- 特殊服務預約等待池
- 高流量活動頁前置排隊入口

## 四、專案範圍

### 4.1 本期建置範圍
本期建置內容如下：

- 使用者加入候補
- 查詢目前順位與前方人數
- 使用者取消候補
- 管理員查詢候補清單
- 管理員分批放行
- 逾時未處理自動失效
- 前端即時接收排隊狀態更新

### 4.2 本期不納入範圍
以下項目不列入第一階段：

- 金流整合
- 多級會員權重
- 複雜推薦或預測演算法
- 微服務全面拆分
- Kafka 作為主流程依賴元件
- 多區域部署

## 五、系統建置原則

本系統之建置原則如下：

1. 主流程優先簡單穩定。
2. 排隊順序必須可預測且不可錯亂。
3. 高流量下優先保護主服務可用性。
4. 即時通知與延伸功能可逐步事件化。
5. 架構需支援後續水平擴充。

## 七、系統架構說明

### 7.1 第一階段架構
第一階段以單體式架構為主，整合 API、排隊邏輯、Redis、資料庫與即時推播。核心目標是先完成可用系統，並確保主流程在高併發情境下仍能穩定運作。

主要流程如下：

1. 使用者發送加入候補請求。
2. API 驗證活動狀態、限流規則與重複加入條件。
3. 系統將資料寫入 Redis 排隊結構。
4. 同步或非同步寫入 PostgreSQL。
5. 回傳排隊結果與目前順位。
6. 透過 WebSocket 將狀態變化推送至前端。

### 7.2 第二階段架構
第二階段導入 Kafka，將通知、審計、分析等附屬流程自主流程中拆出，形成事件驅動架構。此階段目標是進一步提升系統彈性、可維護性與擴充能力。

## 八、功能規劃

### 8.1 使用者功能
- 加入候補
- 查詢目前順位
- 查詢前方等待人數
- 取消候補
- 接收放行通知
- 接收逾時失效通知

### 8.2 管理端功能
- 建立活動
- 開啟候補
- 關閉候補
- 查詢排隊清單
- 分批放行指定數量
- 查詢已放行、已失效、已取消資料

## 九、資料設計規劃

### 9.1 PostgreSQL 資料表

#### queue_event
記錄活動基本資料：

- id
- name
- status
- capacity
- open_time
- close_time
- created_at

#### queue_entry
記錄排隊資料：

- id
- event_id
- user_id
- status
- joined_at
- admitted_at
- expired_at
- cancelled_at

#### queue_audit_log
記錄操作歷程：

- id
- entry_id
- action
- payload
- created_at

### 9.2 Redis Key 規劃

- `queue:{eventId}`：儲存候補順序
- `queue:user:{eventId}:{userId}`：避免重複加入
- `ratelimit:join:{userId}`：限制短時間重複請求
- `admit:lock:{eventId}`：控制放行鎖定

## 十、狀態設計

候補單狀態建議如下：

- WAITING：等待中
- ADMITTED：已放行
- EXPIRED：已逾時失效
- CANCELLED：已取消

狀態流轉說明如下：

1. 使用者加入候補後為 WAITING。
2. 管理員放行後轉為 ADMITTED。
3. 超過處理時間未完成報到則轉為 EXPIRED。
4. 使用者主動取消則轉為 CANCELLED。

## 十一、API 規劃

### 11.1 使用者 API
- `POST /api/events/{eventId}/queue/join`
- `GET /api/events/{eventId}/queue/me`
- `DELETE /api/events/{eventId}/queue/me`

### 11.2 管理端 API
- `POST /api/admin/events`
- `POST /api/admin/events/{eventId}/open`
- `POST /api/admin/events/{eventId}/close`
- `POST /api/admin/events/{eventId}/admit`
- `GET /api/admin/events/{eventId}/queue`

## 十二、即時通知規劃

系統將透過 WebSocket 提供即時通知功能，讓使用者無需反覆輪詢即可收到狀態更新。通知內容包括順位變動、是否已被放行、是否逾時失效等資訊。

建議推播主題如下：

- `/topic/events/{eventId}/queue`
- `/topic/users/{userId}/queue-status`

## 十三、高併發強化策略

為確保系統可承受流量尖峰，規劃採用以下策略：

1. 使用 Redis 作為主要排隊結構。
2. 使用限流機制避免惡意重複請求。
3. 使用快取減少資料庫壓力。
4. 使用分散式鎖避免重複放行或重複入列。
5. 將非必要同步流程延後至第二階段事件化處理。
6. 規劃壓力測試驗證順位正確性與穩定性。

## 十四、Kafka 擴充規劃

第二階段可導入 Kafka，將下列事件輸出供其他服務消費：

- queue-created
- queue-cancelled
- queue-expired
- queue-admitted
- queue-analytics

可拆分之服務包括：

- 通知服務
- 審計服務
- 分析服務

此規劃可避免主 API 同步處理過多附屬任務，提升整體延展性。

## 十七、驗收標準

本專案驗收標準如下：

1. 同一使用者不得重複加入同一活動隊列。
2. 大量同時請求下，順位需保持正確。
3. 管理端放行後，前端須能即時收到更新。
4. 使用者取消後，順位需重新正確計算。
5. 逾時未處理者須自動失效。
6. 系統於高併發情境下不得產生明顯順序錯亂或資料重複。

## 十八、風險與因應

### 18.1 可能風險
- 高流量造成 Redis 或 DB 壓力上升
- 使用者重複提交導致資料不一致
- WebSocket 連線數過高造成推播壓力
- 排程與人工放行同時發生導致狀態競爭

### 18.2 因應方式
- 導入限流與快取
- 使用鎖與唯一鍵保護資料一致性
- 拆分即時推播與主流程
- 對狀態更新進行原子控制

## 十九、結語

QueueFlow 專案適合作為高併發後端系統的入門與進階練習題。其範圍足夠聚焦，能有效練習排隊結構、資料一致性、流量保護、即時通知與事件化架構，並保留後續導入 Kafka 的完整擴充空間。
