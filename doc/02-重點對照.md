# 重點概念對照

---

## 1. Redis vs PostgreSQL：兩者如何分工？

| 面向 | Redis | PostgreSQL |
|------|-------|-----------|
| 儲存位置 | 記憶體（RAM） | 磁碟（持久化） |
| 速度 | 微秒級 | 毫秒級 |
| 資料角色 | 即時排隊結構 | 歷史記錄 |
| 當機後 | 排隊狀態可能消失 | 記錄永遠存在 |
| 查詢能力 | 有限（Key-Value / Sorted Set） | 完整 SQL 查詢 |

**本系統做法**：使用者排名即時讀 Redis（快）；入列事件同步寫 PostgreSQL（正確）。這樣即使 Redis 重啟，DB 仍有歷史記錄，且主要讀寫走 Redis 不會拖慢回應速度。

## 2. Sorted Set score：INCR 序號 vs 毫秒時間戳

| 方式 | INCR 原子序號 ✅ | System.currentTimeMillis() ❌ |
|------|----------------|-------------------------------|
| 原理 | 每次 +1，嚴格遞增 | 取系統時鐘毫秒數 |
| 同毫秒併發 | 不同 score，順序正確 | 相同 score，Sorted Set 改用 userId 字典序排序，**FIFO 失效** |
| 實作位置 | `RedisQueueStore.nextSequence()` | — |

```java
// RedisQueueStore.java
public long nextSequence(Long eventId) {
    Long seq = redis.opsForValue().increment(String.format(SEQ_KEY, eventId));
    return seq == null ? System.currentTimeMillis() : seq;
}
```

`INCR` 指令本身是原子操作，不會重複，也不受時鐘影響。

## 3. Lua Script vs 多步驟 Redis 操作

**問題場景**：加入排隊需要兩步驟：
1. 檢查 user key 是否存在（防重複）
2. SETEX user key + ZADD queue

若步驟 1 到步驟 2 之間另一個請求同時執行，兩人都能通過檢查 → 同一使用者加入兩次。

| 方式 | 特性 |
|------|------|
| 分兩次 Redis 操作 | 步驟之間有競態條件，不安全 |
| Lua Script（單次執行）✅ | Redis 保證 Lua script 為原子操作，執行期間不切換到其他命令 |

```lua
-- JOIN_SCRIPT（RedisQueueStore.java 內）
local userKey  = KEYS[1]
local queueKey = KEYS[2]
if redis.call('EXISTS', userKey) == 1 then return 0 end  -- 已在隊中，拒絕
redis.call('SETEX', userKey, ARGV[3], '1')               -- 標記使用者
redis.call('ZADD', queueKey, ARGV[2], ARGV[1])           -- 加入排序集
return 1                                                  -- 成功
```

## 4. 分散式鎖 vs Java synchronized

| 面向 | Java `synchronized` | Redis SET NX（分散式鎖）✅ |
|------|---------------------|--------------------------|
| 作用範圍 | 同一 JVM 內的執行緒 | 跨服務、跨機器 |
| 水平擴展 | 無法跨多個應用實例 | 所有實例共享同一把鎖 |
| 鎖失效 | JVM 崩潰即釋放 | TTL 到期自動釋放（防死鎖） |
| 本系統用途 | 不使用 | `admit:lock:{eventId}` 防止管理端重複放行 |

```java
// EventService.java
if (!redisStore.tryAcquireLock(eventId, 10L)) {
    throw AppException.conflict("放行操作正在進行中，請稍後再試");
}
try {
    // 放行邏輯...
} finally {
    redisStore.releaseLock(eventId);   // 務必在 finally 釋放
}
```

## 5. WebSocket vs HTTP 輪詢

| 面向 | HTTP 輪詢 | WebSocket ✅ |
|------|-----------|-------------|
| 連線模式 | 每次請求建立新連線 | 一次握手，持續雙向通道 |
| 延遲 | 取決於輪詢間隔（通常 1–5s） | 近即時（< 100ms） |
| 伺服器負載 | 大量重複請求 | 推播，只在有資料時發送 |
| 實作複雜度 | 簡單 | 需要 STOMP broker + SockJS |

本系統使用 STOMP over SockJS：
- SockJS：在 WebSocket 不可用時降級為 HTTP long-polling
- STOMP：提供主題（`/topic/...`）訂閱語意，管理員和使用者各訂不同主題

## 6. AppException 工廠方法 vs 手動建構 HTTP 回應

```java
// ❌ 傳統做法：每個錯誤自行建構 ResponseEntity
return ResponseEntity.status(409).body(new ErrorResponse("重複請求"));

// ✅ 本系統做法：AppException 靜態工廠 → GlobalExceptionHandler 統一轉換
throw AppException.conflict("您已在候補名單中");
```

`GlobalExceptionHandler` 攔截所有 `AppException`，讀取 `.getStatus()` 組出 `ApiResponse<Void>`。新增錯誤類型只需呼叫現有工廠方法，不動 Handler。

| 工廠方法 | HTTP 狀態碼 |
|---------|------------|
| `notFound()` | 404 |
| `badRequest()` | 400 |
| `conflict()` | 409 |
| `tooManyRequests()` | 429 |

## 7. @Transactional 範圍設計

```
EventService   → class-level @Transactional
                 所有方法都在同一個交易邊界，符合管理操作的一致性需求

QueueService   → method-level，各自獨立
  joinQueue    → @Transactional          (寫入)
  cancelQueue  → @Transactional          (寫入)
  getPosition  → @Transactional(readOnly=true)  (唯讀，省略 flush/dirty-check)
```

`readOnly=true` 讓 Hibernate 跳過 dirty-checking 與 flush，查詢效能較佳；同時告知資料庫可使用讀副本（若有配置）。
