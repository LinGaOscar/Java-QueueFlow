# 周邊機制說明

---

## 1. 限流（Rate Limiting）

**目的**：防止同一使用者在短時間內重複按加入，避免洗版攻擊。

**實作**：`RedisQueueStore.isRateLimited(userId)`

```java
public boolean isRateLimited(String userId) {
    String key = String.format("ratelimit:join:%s", userId);
    Boolean set = redis.opsForValue().setIfAbsent(key, "1", Duration.ofSeconds(5));
    return Boolean.FALSE.equals(set);  // false = key 已存在 = 被限流
}
```

**機制說明**：
- `SET NX EX 5`：若 key 不存在才寫入，並設 5 秒 TTL
- 第一次請求：寫入成功 → 放行
- 5 秒內第二次請求：key 已存在，`setIfAbsent` 回傳 `false` → 拒絕（429）
- 5 秒後：TTL 到期，key 消失，使用者可再次嘗試

**注意**：此限流基於 `userId` 字串，前端若讓使用者自填 userId，惡意使用者可以換 userId 繞過。生產環境應改為 IP 或 Session 層級的限流。

---

## 2. 防重複入列

**目的**：確保同一使用者在同一活動中最多只有一筆 WAITING 記錄。

**實作**：Lua Script 原子執行（詳見 `02-重點對照.md` 第 3 節）

```
queue:user:{eventId}:{userId}  → 標記鍵，TTL = 86400 秒（1 天）
```

**取消或放行後**：`removeFromQueue()` 同時刪除 Sorted Set 成員與 user key，讓使用者可以重新入列。

---

## 3. 逾時排程（QueueExpiryScheduler）

**目的**：活動關閉後，留在 DB 的 WAITING 記錄不會自動消失，需要排程清除。

**實作**：`@Scheduled(fixedDelay = 60_000)`，每完成一次後等待 60 秒再執行。

```java
// 查詢條件：活動已 CLOSED，且 entry 仍為 WAITING
List<QueueEntry> stale = entryRepository.findWaitingEntriesForClosedEvents();
```

**執行順序**：
1. 找出所有符合條件的 entry（跨多個關閉活動）
2. 逐筆設為 EXPIRED，寫 audit log
3. 從 Redis 移除（防止殘留）
4. 推播 `EXPIRED` 給各使用者
5. 按 eventId 去重，推播各活動的新排隊大小

**關鍵語意**：只清除**活動已關閉後**的殘留記錄，不干涉活動仍 OPEN 時正在等候的使用者。

---

## 4. 稽核日誌（Audit Log）

**目的**：留存完整操作歷程，方便事後查驗或爭議處理。

**表格**：`queue_audit_log`（entry_id, action, payload, created_at）

| action | 觸發時機 |
|--------|---------|
| `JOIN` | 使用者加入排隊（joinQueue） |
| `ADMITTED` | 管理員放行（admitQueue） |
| `CANCELLED` | 使用者取消（cancelQueue） |
| `EXPIRED` | 排程清除（expireStaleEntries） |

目前 payload 欄位均為 `null`，預留供未來記錄額外資訊（如管理員 ID、IP）。

## 5. Schema 自動建立

**設定**：`spring.sql.init.mode=always` + `spring.jpa.defer-datasource-initialization=true`

**流程**：

```
Spring Boot 啟動
  └─ JPA EntityManagerFactory 初始化完成（defer 確保順序）
      └─ 執行 src/main/resources/schema.sql
          └─ CREATE TABLE IF NOT EXISTS ...（冪等，重複啟動安全）
```

**為何不用 `ddl-auto=create`？**  
`create` 每次啟動會刪舊表再建新表，資料全毀。`ddl-auto=none` + 自訂 SQL 讓開發者完全掌控 schema 變更。

## 6. 全域例外處理

**設定**：`GlobalExceptionHandler`（`@RestControllerAdvice`）

```
任意 Controller 拋出例外
  ├─ AppException      → status = e.getStatus()，message = e.getMessage()
  ├─ NoResourceFoundException → 404（靜態資源不存在，Spring Boot 3.2+ 變更行為）
  └─ Exception（其他）  → 500，記錄 log.error
```

所有回應統一格式：

```json
{ "success": false, "data": null, "message": "錯誤描述" }
```

前端可一律檢查 `r.success` 判斷是否成功，不須逐一判斷 HTTP 狀態碼。

## 7. 前端注意事項（教學提醒）

### XSS 防護
所有由使用者輸入或 API 回傳的字串，插入 `innerHTML` 前必須經過 `esc()` 跳脫：

```javascript
function esc(s) {
  return String(s)
    .replace(/&/g,'&amp;')
    .replace(/</g,'&lt;')
    .replace(/>/g,'&gt;')
    .replace(/"/g,'&quot;');
}
// 用法：element.innerHTML = `<div>${esc(ev.name)}</div>`
```

### inline onclick 命名禁忌
`onclick="fn()"` 的作用域鏈包含 `with(document)`，以下名稱會與瀏覽器 DOM API 衝突，應避免使用：

- `createEvent` → 與 `document.createEvent()` 衝突
- `open` → 與 `window.open()` 衝突
- `close` → 與 `window.close()` 衝突
- `focus`, `blur`, `submit` 等 HTML 元素方法

**本系統已將 `createEvent` 改名為 `submitNewEvent` 以修復此問題。**

### 靜態資源更新
修改 `src/main/resources/static/` 內的 HTML/JS 後，需要重啟 Spring Boot 伺服器才能服務新版本（`mvn spring-boot:run` 從 `target/classes/static` 服務，非即時熱更新）。
