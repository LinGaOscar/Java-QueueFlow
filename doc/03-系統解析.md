# 系統流程解析

---

## 一、QueueEntry 狀態機

每筆 `queue_entry` 記錄有明確的狀態流轉，不可逆：

```
                  ┌─────────────────────────────────────────┐
                  │                  WAITING                 │
                  │  （Redis Sorted Set 中有記錄）            │
                  └──────┬──────────────┬──────────────┬────┘
                         │              │              │
               管理員放行  │   使用者取消  │   活動關閉後  │
                         │   (DELETE)   │   排程清除   │
                  ┌──────▼───┐  ┌───────▼──┐  ┌──────▼──┐
                  │ ADMITTED │  │CANCELLED │  │ EXPIRED │
                  └──────────┘  └──────────┘  └─────────┘
```

每次狀態變更都寫入 `queue_audit_log`，action 分別為 `ADMITTED` / `CANCELLED` / `EXPIRED`。

## 二、使用者加入排隊（joinQueue）完整流程

```
[前端] POST /api/events/{eventId}/queue/join?userId=xxx
           │
[QueueService.joinQueue()]
    ①  isRateLimited(userId)
       ├─ Redis SET NX ratelimit:join:{userId}  EX 5s
       └─ key 已存在 → 拒絕（429 Too Many Requests）

    ②  查 PostgreSQL 確認 eventId 存在且 status = OPEN
       └─ 不存在或非 OPEN → 拒絕（404 / 400）

    ③  nextSequence(eventId)
       └─ Redis INCR queue:seq:{eventId}  → 取得唯一遞增 score

    ④  tryJoin(eventId, userId, score, 86400s)  [Lua Script]
       ├─ EXISTS queue:user:{eventId}:{userId}
       │    = 1 → 已在隊中，return 0（409 Conflict）
       ├─ SETEX queue:user:{eventId}:{userId} 86400 "1"
       └─ ZADD queue:{eventId} {score} {userId}

    ⑤  @Transactional 同步寫入 PostgreSQL
       ├─ INSERT queue_entry (status=WAITING)
       └─ INSERT queue_audit_log (action=JOIN)

    ⑥  getPosition(eventId, userId)
       └─ Redis ZRANK queue:{eventId} {userId} + 1

    ⑦  notifyQueueUpdate(eventId, queueSize)
       └─ WebSocket → /topic/events/{eventId}/queue

[回應] { position: N, ahead: N-1, status: "WAITING" }
```

## 三、管理員分批放行（admitQueue）完整流程

```
[前端] POST /api/admin/events/{eventId}/admit   body: {"count": N}
           │
[EventService.admitQueue()]
    ①  count ≤ 0 → 拒絕（400）

    ②  tryAcquireLock(eventId, 10s)
       └─ Redis SET NX admit:lock:{eventId} EX 10
          失敗 → 409 Conflict（另一個放行正在進行）

    ③  getTopN(eventId, count)
       └─ Redis ZRANGE queue:{eventId} 0 N-1
          取排名最前 N 個 userId

    ④  for each userId:
       ├─ DB 查詢 WAITING 狀態的 QueueEntry（避免多次入列時 non-unique）
       ├─ entry.status = ADMITTED, admittedAt = now()
       ├─ INSERT queue_audit_log (action=ADMITTED)
       ├─ Redis ZREM + DEL user key（removeFromQueue）
       └─ WebSocket → /topic/users/{userId}/queue-status  { status: "ADMITTED" }

    ⑤  notifyQueueUpdate(eventId, newQueueSize)
       └─ WebSocket → /topic/events/{eventId}/queue

    ⑥  finally: releaseLock(eventId)
       └─ Redis DEL admit:lock:{eventId}

[回應] 實際放行人數（int）
```

## 四、WebSocket 即時推播架構

```
             訂閱 /topic/events/{eventId}/queue
使用者 A ────┐
使用者 B ────┤──► STOMP Broker (記憶體)
管理員  ─────┘
             訂閱 /topic/users/{userId}/queue-status

當管理員放行：
  EventService
    ├─ notifyUserStatus("user-001", "ADMITTED")
    │    └─ simpMessagingTemplate.convertAndSend(
    │         "/topic/users/user-001/queue-status", {status:"ADMITTED"})
    │         → 只有訂閱此 topic 的 user-001 收到
    └─ notifyQueueUpdate(eventId, newSize)
         └─ simpMessagingTemplate.convertAndSend(
              "/topic/events/1/queue", {queueSize:99})
              → 所有在等待的人更新顯示
```

前端收到 `ADMITTED` 後彈出「恭喜入場」覆蓋層；收到 `EXPIRED` 後顯示 toast 提示。

## 五、查詢排隊位置（getPosition）的雙軌設計

```
getPosition(eventId, userId)
    │
    ├─① Redis ZRANK 有記錄？
    │     → 返回 (rank+1, rank, "WAITING")
    │
    └─② Redis 無記錄（已放行 / 已取消 / 已逾時）
          → DB: findFirstByEventIdAndUserIdOrderByJoinedAtDesc
            → 找最新一筆，返回其終態 status
```

這個設計讓前端在使用者已結束排隊後，還能查到最終狀態，不會顯示「查無記錄」。

## 六、活動生命週期

```
建立 (PENDING) ──► 開放 (OPEN) ──► 關閉 (CLOSED)
  管理員 POST /events   POST /open     POST /close

OPEN 期間：
  - 使用者可 join / cancel
  - 管理員可 admit（逐批放行）
  - WebSocket 推播排隊大小

CLOSED 之後：
  - QueueExpiryScheduler（每分鐘）掃描
  - 找出 CLOSED 活動的殘留 WAITING 記錄
  - 批次改為 EXPIRED，推播通知使用者
```
